1.1 Role in the OPS Architecture

The OPS Architecture [OPS-D3.2.3] provides a high-level view of the Open PHACTS framework, with a detailed specification of the identity and vocabulary services given in [BEG+2012]. The Open PHACTS RDF guidelines [HWZ+2012] specify the vocabularies that are recommended for publishing datasets for use in the OPS system. This specification defines the format of identity mapping linksets that relate the concepts[1] in the various datasets loaded in OPS. The provenance and metadata guidelines about the linksets given in this specification should be used in conjunction with the equivalent statements in the RDF publication guidelines [HWZ+2012].[d]

The identity mapping linksets will predominantly be used as the means to import data to the Identity Mapping Service (IMS) through the interface defined in Section 4.4.2 of [BEG+2012]. The mapping specification will also form the basis of the information model returned through the Identity Mapping Service query interface defined in Section 4.4.1 of [BEG+2012].

2.2. Vocabulary of Interlinked Datasets (VoID)

The Vocabulary of Interlinked Datasets (VoID) [VOID] is a W3C interest group note that specifies an RDF schema for describing the metadata about a dataset and its relationship with other datasets. The vocabulary builds upon existing metadata vocabularies, e.g. Dublin Core [DC].

The VoID specification [VOID] provides the following definitions:

Dataset: “A set of RDF triples that are published, maintained or aggregated by a single provider.”

Link: “An RDF triple whose subject and object are described in different datasets.”

Linkset: “A collection of such RDF links between two datasets.”

The VoID specification enables a separation between (1) the datasets involved in a link, and (2) who publishes the link.

Note that a VoID linkset is defined to link two datasets via a single link predicate (void:linkPredicate). As such, there can exist multiple linksets relating the same pair of datasets. The following are examples[f] of s[g]ome of the multiple linksets that can exist between datasets ChemSpider and ChEMBL:

  1. ChemSpider cs:extUrl ChEMBL as published in the ChemSpider dataset
  2. ChemSpider cs:... ChEMBL as published in the ChemSpider dataset
  3. ChEMBL :csid ChemSpider as published in the ChEMBL dataset
  4. ChEMBL :somePredicate ChemSpider as published by some third party

2.2.1. Vocabulary for Data and Dataset Provenance (voidp)

Defined as an extension to VoID, the vocabulary for data and dataset provenance (voidp) [VOIDP] is a vocabulary for defining provenance relationships of data and datasets. The vocabulary focuses on four specific pieces of provenance information:

“for a piece of data, x :

  1. when was x derived,
  2. how was x derived,
  3. what data had been used to derive x,
  4. who carried out the transformations that resulted in the current value of x.” 

[VOIDP]

These are a subset of the information that needs to be captured for the Open PHACTS linksets.

The vocabulary provides its own predicates for capturing these provenance relationships, rather than reusing existing vocabularies for capturing provenance metadata, e.g. the open provenance model or Provenance Authoring and Versioning ontology [CWW+2008, PAV]. There is now also a W3C group working on a provenance ontology [BCG+2011].

3. Anatomy of a Mapping Specification (normative)

A data instance mapping is a claim that an identifier in one dataset is related in some way to an identifier in another dataset. The mapping specification captures the type of relationship and the metadata associated with the mapping, e.g. provenance information about who created the mapping and the specific versions of the datasets related.

Example linksets are given in Appendix A. Example Linksets. The example linkset in Appendix A.1 presents a simple case where a set of ConceptWiki resources are mapped to a set of ChemSpider resources. Appendix A.2 presents a more complex example which relates the ChemSpider dataset to the ChEMBL compounds dataset as represented by the Chem2Bio2RDF conversion of ChEMBL version 2 [CDJ+2010].

3.1. Representing a Mapping

A mapping must be represented as a VoID link, i.e. it is an RDF triple that relates an identifier in one dataset with an identifier in another dataset with some predicate which provides the meaning of the mapping.

The mapping predicate captures the way in which the two identifiers are related. The mapping must[h] respect[i] the semantics of the relationship, i.e. the owl:sameAs relationship must only be used when the two identifiers are completely interchangeable.

Halpin et al [HHM+2010] propose a mapping relationship ontology with 8 levels of equivalence:

  1. Identical (equivalent to owl:sameAs): always interchangeable
  2. Claims Identical: inverse does not hold
  3. Matches (equivalent to skos:exactMatch): share many properties, interchangeable in most cases
  4. Claims Matches: inverse does not hold
  5. Similar (equivalent to skos:closeMatch): share some properties, not interchangeable
  6. Claims Similar: inverse does not hold
  7. Related (equivalent to skos:related): distinct things with an association
  8. Claims Related (equivalent to rdfs:seeAlso)

It is recommended that these relationships are used when more suitable domain specific relationships do not exist.[j]

Examples

The following example declares that the ChemSpider α-Ketoisovaleric acid

concept with the ID 48[2] shares many properties with the ChEMBL 3-Methyl-2-oxobutanoic acid concept with the Chem2Bio2RDF ChEMBL ID 289013[3]. The relationship is drawn from ChemSpider based on the compounds sharing the same structure based on their InChI Keys[4].

chemspider:48 skos:exactMatch chembl:289013 .

[k]

The following example declares that the Brenda enzyme alcohol dehydrogenase has the UniProt EC number 1.1.1.1.

brenda:1.1.1.1 brenda:has_ec_number[l] uniprotEnzyme:1.1.1.1 .

Context

For different tasks it is appropriate to assert different levels of equivalence between the same identifiers. For example, consider …[m] . In this context it is appropriate to map as equivalent the identifiers of compounds that share the same structure but differ in their stereo-chemistry. At the same time the mappings should distinguishing between gene identifiers and protein identifiers. In other circumstance the inverse should hold, i.e. the compounds should be distinguished whilst the genes and proteins should be considered equivalent.

A mapping should have context associated with it to support the use of different mappings based on the task being performed. Within the OPS framework, common contexts will be defined using profile specifications [PS2012]. A mapping link should be associated with one or more profiles using the ops:linksHoldInProfile predicate.

Example (Informative)

The following RDF statement states that the :chemspider_chembl link holds in the chemistry profile (ops:chemistry).

:chemspider_chembl ops:linksHoldInProfile ops:chemistry .

3.2. Metadata

3.2.1. Dataset Metadata

A linkset associates two datasets via a single mapping predicate. For each dataset in the linkset, the following metadata must[n] be included:

Type: 

  1. Declare that we have a dataset using void:dataset.

Version:

  1. If the dataset has the notion of publication versions then this must be captured using the pav:Version property. [o]

Source provenance:

  1. If the dataset is derived from another data source, i.e. the knowledge content has changed, then the following properties should be given:
  1. pav:derivedFrom to capture the relationship to the original source.
  2. pav:derivedOn to capture when the derivation was made.
  3. pav:derivedBy to capture who made the derivation. The object of this triple should use an ORCID[p] ID [ORCID].[q]
  1. If the dataset has been retrieved from another location, i.e. it is used in exactly the same form, then the following properties should be given:
  1. pav:retrievedFrom to capture the location of the original source.
  2. pav:retrievedOn to capture when the retrieval was made.
  3. pav:retrievedBy to capture who made the retrieval. The object of this triple should use an ORCID ID [ORCID].
  1. If the dataset is imported from another data source, i.e. the knowledge content is unchanged but the format of the data has been changed (e.g. XML to RDF transformation), then the following properties should be given:
  1. pav:importedFrom to capture the location of the original source.
  2. pav:importedOn to capture when the import was made.
  3. pav:importedBy to capture who made the import. The object of this triple should use an ORCID ID [ORCID].
  1. If the dataset is accessed from its original location for creating the dataset, then the following properties should be given:
  1. pav:sourceAccessedAt to capture the location accessed.
  2. pav:sourceAccessedOn to capture the date of access.
  3. pav:sourceAccessedBy to capture who made the access. The object of this triple should use an ORCID ID [ORCID]. 

Namespace:

  1. The URI namespace for the resources being linked is declared using the void:uriSpace predicate[5].

Resource type:

  1. The type of the resource being linked is declared with the void:class[r] predicate.

In order to support applications using the linkset to understand and present the data, the following additional metadata should be included:

Title:

  1. The title of the dataset using dcterms:title predicate.[s]

Description:

  1. A description of the dataset using dcterms:description predicate.

Links to further information:

  1. A link to the homepage for the dataset foaf:hompage[6].
  2. Other pages that discuss the dataset may be included with the foaf:page relationship

Data source:

  1. If the dataset is a subset of another dataset, then this is captured with the void:subset predicate. The metadata that is common between these datasets can be declared for the parent only.

Other metadata that can be associated with a dataset may be included, see the RDF publishing guidelines [HWZ+2012], the VoID specification [VOID], and the provenance, authorship and versioning vocabulary [PAV] for additional metadata that may be incorporated. The dataset may also have a provenance graph associated with it providing more detailed information about the creation, authorship, and derivation of the dataset.

Example: Dataset metadata for ChemSpider

:chemspider

    a void:dataset ;

    pav:importedOn "2012-02-22"^^xsd:date ;

    pav:importedFrom <http://rdf.chemspider.com/> ;

    pav:importedBy <http://www.conceptwiki.org/> ;

    void:uriSpace "http://rdf.chemspider.com/" ;

    void:class <XXX[t]> ;

    dcterms:title "ChemSpider" ;

    dcterms:description "A chemical structure database." ;

    foaf:homepage <http://rdf.chemspider.com/> ;

    foaf:page <http://www.chemspider.com/> .

Note that in the example the homepage has been set to the RDF representation of ChemSpider as that is the dataset that is linked in the linkset. The normal web interface has been given as an alternative web page. As the ChemSpider database does not have version numbers, the pav:importedOn property is used to state the version of ChemSpider that the linkset was created against. For a dataset such as ChEMBL which does have versions, this would be stated as

:chembl_compounds

    a void:dataset ;

    void:subset :chembl ;

    void:uriSpace "http://chem2bio2rdf.org/chembl/resource/chembl_compounds/" ;

    void:class <http://chem2bio2rdf.org/chembl/resource/chembl_compounds> ;

    dcterms:title "Chem2Bio2RDF ChEMBL Compounds" ;

    dcterms:description "The compounds subset of the Chem2Bio2RDF conversion of the ChEMBL database." ;

    foaf:homepage <http://chem2bio2rdf.org/chembl/resource/chembl_compounds/> .

:chembl

    a void:dataset ;

    pav:versionNumber "02"^^xsd:integer ;

    pav:retrievedFrom <http://cheminfov.informatics.indiana.edu:8080/download/chembl.n3.gz> ;

    pav:retrievedOn "2011-11-30"^^xsd:date ;

    pav:retrieveBy [ a foaf:Person ; foaf:name "Antonis Loizou"@en . ] ;

    foaf:homepage <http://chem2bio2rdf.org/chembl/> ;

    foaf:page <https://www.ebi.ac.uk/chembl/> ;

    dcterms:title "Chem2Bio2RDF ChEMBL" ;

    dcterms:description "The Chem2Bio2RDF conversion of the ChEMBL database." .

3.2.2. Linkset[u] Metadata

A linkset is itself a dataset, and as such should provide the metadata one would associate with a dataset, as published in the Open PHACTS RDF Guidelines [HWZ+2012]. The following metadata must be included:

Type: 

  1. Declare that we have a linkset using void:linkset.

License:

  1. The license under which the linkset is published must be stated using the dcterms:license property.

Linkset provenance[7]:

  1. If the linkset is a subset of another dataset, then this is captured with the void:subset predicate.
  2. The linkset authorship, i.e. the agent that generated the intellectual knowledge, is captured with the following properties:
  1. pav:authoredBy to capture who authored the links. The object of this triple should use an ORCID[p] ID [ORCID].
  2. pav:authoredOn to capture when the links were generated.
  1. The linkset creator, i.e. the entity responsible for converting the links to the linkset format, is captured with the following properties:
  1. pav:createdBy to capture who converted the links into the linkset format. The object of this triple should use an ORCID ID [ORCID].
  2. pav:createdOn to capture when the linkset was made.

Link metadata:

  1. If the link predicate is not symmetric, then the void:subjectsTarget and void:objectsTarget must be used. Otherwise the super property void:target can be declared for both datasets instead.
  2. The link predicate is declared using void:linkPredicate.
  3. The number of links in the linkset is declared using void:triples. This allows for validation that the entire linkset has been successfully loaded.

In order to support applications using the linkset to understand and present the data, the following additional metadata should be included:

Title:

  1. The title of the linkset using dcterms:title predicate.

Description:

  1. A description of the linkset using dcterms:description predicate.

Note that for linksets there is unlikely to be a web page directly dedicated to the linkset, as such that part of the dataset metadata is not stated as a requirement.

Example: ChemSpider to Chem2Bio2RDF ChEMBL Compounds Linkset Metadata

:chemspider_chembl

    a void:Linkset ;

    dcterms:license <http://www.opendatacommons.org/licenses/odbl/> ;

    void:subset :chemspider ;

    pav:authoredOn "2012-02-22"^^xsd:date ;

    pav:authoredBy <http://www.chemspider.com/> ;

    pav:createdOn "2012-03-01"^^xsd:date ;

    pav:createdBy :alasdair ;

    void:target :chembl_compounds ;

    void:target :chemspider ;

    void:linkPredicate skos:exactMatch ;

    void:triples 1024305 ;

    dcterms:title "ChemSpider to ChEMBL Compounds" ;

    dcterms:description "Linkset relating ChemSpider compounds to ChEMBL compounds using the Chem2Bio2RDF encoding of ChEMBL." .

3.3.3. Provenance[v]

Details on how the links are created, i.e. the process, algorithm, and software, can be provided in a separate named graph using the provenance ontology [BCG+2011]. Details of this approach will be further evolved in future versions of this specification and will build on the approach being developed for nanopublications [SCB+2012].

4. Exchanging Mapping Documents (normative)

It is anticpated that links between datasets will not be created and shared in isolation, i.e. multiple links relating concepts in two datasets with the same predicate will be created at the same time using the same mechanism. A collection[w] of links that meet the following requirements can be shared as a single VoID linkset:

  1. The resulting document must be a valid VoID linkset:
  1. The subject of all of the mapping triples are from the same dataset
  2. The object of all the mapping triples are from the same dataset
  3. The predicate of all the mapping triples are the same
  1. The metadata of all the mapping triples is the same

The interpretation must be that each link in the linkset is a mapping with the metadata given in the heading of the VoID linkset. The linkset should include the metadata specified in Section 3.2.

When links are shared as a VoID linkset, the document must contain the void:triples predicate declaring the number of links contained in the linkset. This supports verification that all triples are imported correctly. Examples of VoID linksets can be found in Appendix A.
