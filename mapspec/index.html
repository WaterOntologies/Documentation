This is a specification for the links between datasets in Open PHACTS and their exchange format.

Intended audience

These mapping specifications are intended for data providers who want to interlink their RDF published data on the Open PHACTS platform with other RDF datasets.

Introduction

Role in the OPS Architecture

The OPS Architecture [[OPS-ARCH]] provides a high-level view of the Open PHACTS framework, with a detailed specification of the identity and vocabulary services given in [[OPS-IV]]. The Open PHACTS RDF guidelines [[OPS-RDF]] specify the vocabularies that are recommended for publishing datasets for use in the OPS system. This specification defines the format of identity mapping linksets that relate the concepts in the various datasets loaded in OPS. The provenance and metadata guidelines about the linksets given in this specification should be used in conjunction with the equivalent statements in the RDF publication guidelines [[OPS-RDF]].

The identity mapping linksets will predominantly be used as the means to import data to the Identity Mapping Service (IMS) [[IMS]] through the interface defined in Section 4.4.2 of [[OPS-IV]]. The mapping specification will also form the basis of the information model returned through the Identity Mapping Service query interface defined in Section 4.4.1 of [[OPS-IV]].

Open PHACTS RDF Guidelines

The Open PHACTS project has produced a set of guidelines aimed at data providers for publishing their data within the OPS system [[!OPS-RDF]]. The guide provides details about data licenses and publishing a dataset description using VoID. This mapping specification exploits these existing VoID descriptions.

RDF Syntax Used for Examples

All examples in this document are written in the Turtle RDF syntax [[TURTLE]]. Throughout the document, the following namespaces are used:

@prefix dcterms: <http://purl.org/dc/terms/> .
@prefix foaf: <http://xmlns.com/foaf/0.1/> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix pav: <http://purl.org/pav/2.0/> .
@prefix prov: <http://www.w3.org/ns/prov-o/> . 
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix skos: <http://www.w3.org/2004/02/skos/core#> .
@prefix void: <http://rdfs.org/ns/void#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .

Furthermore, we assume that the empty prefix is bound to the base URL of the current file like this:

@prefix : <#> .

This allows us to quickly mint new identifiers in the local namespace.

Related Standards

Vocabulary of Interlinked Datasets (VoID)

The Vocabulary of Interlinked Datasets (VoID) [[!VOID]] is a W3C interest group note that specifies a vocabulary for describing the metadata about a dataset and its relationship with other datasets. The vocabulary builds upon existing metadata vocabularies, e.g. Dublin Core Terms [[DCTERMS]].

The VoID specification [[VOID]] provides the following definitions:

Dataset
A set of RDF triples that are published, maintained or aggregated by a single provider.
Link
An RDF triple whose subject and object are described in different datasets.
Linkset
A collection of such RDF links between two datasets.

The VoID specification enables a separation between (1) the datasets involved in a link, and (2) who publishes the link.

Note that a VoID linkset is defined to link two datasets via a single link predicate (void:linkPredicate). As such, there can exist multiple linksets relating the same pair of datasets. The following are examples of some of the multiple linksets that can exist between datasets ChemSpider and ChEMBL:

  • ChemSpider cs:extUrl ChEMBL as published in the ChemSpider dataset
  • ChemSpider cs:... ChEMBL as published in the ChemSpider dataset
  • ChEMBL :csid ChemSpider as published in the ChEMBL dataset
  • ChEMBL :somePredicate ChemSpider as published by some third party

Vocabulary for Data and Dataset Provenance (voidp)

Defined as an extension to VoID, the vocabulary for data and dataset provenance (voidp) [[VOIDP]] is a vocabulary for defining provenance relationships of data and datasets. The vocabulary focuses on four specific pieces of provenance information:

"for a piece of data, x :
  • when was x derived,
  • how was x derived,
  • what data had been used to derive x,
  • who carried out the transformations that resulted in the current value of x." [[VOIDP]]

These are a subset of the information that needs to be captured for the Open PHACTS linksets.

The vocabulary provides its own predicates for capturing these provenance relationships, rather than reusing existing vocabularies for capturing provenance metadata, e.g. the open provenance model [[OPM]] or Provenance Authoring and Versioning ontology [[PAV]]. There is now also a W3C group working on a provenance ontology [[PROV-O]].

Anatomy of a Mapping Specification

A data instance mapping is a claim that an identifier in one dataset is related in some way to an identifier in another dataset. The mapping specification captures the type of relationship and the metadata associated with the mapping, e.g. provenance information about who created the mapping and the specific versions of the datasets related.

Example linksets are given in Appendix A. The example linkset in Appendix A.1 presents a simple case where a set of ConceptWiki resources are mapped to a set of ChemSpider resources. Appendix A.2 presents a more complex example which relates the ChemSpider dataset to the ChEMBL compounds dataset as represented by the Chem2Bio2RDF conversion of ChEMBL version 2 [[Chem2Bio2RDF]].

Representing a Mapping

A mapping must be represented as a VoID link, i.e. it is an RDF triple that relates an identifier in one dataset with an identifier in another dataset with some predicate which provides the meaning of the mapping.

The mapping predicate captures the way in which the two identifiers are related. The mapping should respect the semantics of the relationship, i.e. the owl:sameAs relationship must only be used when the two identifiers are completely interchangeable.

Halpin et al [[Halpin2010]] propose a mapping relationship ontology with eight levels of equivalence:

  • Identical (equivalent to owl:sameAs): always interchangeable
  • Claims Identical: inverse does not hold
  • Matches (equivalent to skos:exactMatch): share many properties, interchangeable in most cases
  • Claims Matches: inverse does not hold
  • Similar (equivalent to skos:closeMatch): share some properties, not interchangeable
  • Claims Similar: inverse does not hold
  • Related (equivalent to skos:related): distinct things with an association
  • Claims Related (equivalent to rdfs:seeAlso)

Example

The following example declares that the ChemSpider α-Ketoisovaleric acid concept with the CSID 48 shares many properties with the ChEMBL 3-Methyl-2-oxobutanoic acid concept with the Chem2Bio2RDF ChEMBL ID 289013. The relationship is drawn from ChemSpider based on the compounds sharing the same structure based on their InChI Keys (in this case the compound has the same InChI Key QHKABHOOEWYVLI-UHFFFAOYSA-N in both data sets.).

@prefix skos: <http://www.w3.org/2004/02/skos/core#> .
<http://rdf.chemspider.com/48> skos:exactMatch <http://chem2bio2rdf.org/chembl/resource/chembl_compounds/289013> .

Metadata

The metadata associated with a link is essential for enabling its reuse by others. It enables a consumer of the link to understand who claimed the link, under what circumstances, and which (if any) tools were used to generate the link (e.g. [[SILK]]).

The metadata is split into metadata about the dataset and the metadata about the links themselves. Note that the dataset metadata should be provided by the dataset provider as part of the dataset publishing process [[OPS-RDF]], although it can be augmented for the purposes of providing additional information about the link.

Dataset Metadata

A linkset associates two datasets via a single mapping predicate. For each dataset in the linkset, the following metadata must be included:

Type

  • Declare that we have a dataset using void:dataset.

License

  • Provide details of the licence under which the dataset is published using the dcterms:license property.

Version

  • If the dataset has the notion of publication versions then this must be captured using the pav:Version property.

Source Provenance

  • If the dataset is derived from another data source, i.e. the knowledge content has changed, then the following properties should be given:
    • pav:derivedFrom to capture the relationship to the original source.
    • pav:derivedOn to capture when the derivation was made.
    • pav:derivedBy to capture who made the derivation. The object of this triple is a URI, currently we recommend using FOAF [[FOAF]].
  • If the dataset has been retrieved from another location, i.e. it is used in exactly the same form, then the following properties should be given:
    • pav:retrievedFrom to capture the location of the original source.
    • pav:retrievedOn to capture when the retrieval was made.
    • pav:retrievedBy to capture who made the retrieval. The object of this triple is a URI, currently we recommend using FOAF [[FOAF]].
  • If the dataset is imported from another data source, i.e. the knowledge content is unchanged but the format of the data has been changed (e.g. XML to RDF transformation), then the following properties should be given:
    • pav:importedFrom to capture the location of the original source.
    • pav:importedOn to capture when the import was made.
    • pav:importedBy to capture who made the import. The object of this triple is a URI, currently we recommend using FOAF [[FOAF]].
  • If the dataset and its description are accessed from their original location, then the no properties are required. However it is good practice to include the following:
    • Details of who is publishing the dataset using the dcterms:publisher property.
    • Details of when the dataset was created using the dcterms:created property.
    • When appropriate, details of when the last update was made using the dcterms:modified property.

We currently recommend the use of FOAF to capture the person or entity that made a dataset available. In the future, we anticipate the use of ORCID Identifiers [[ORCID]].

Namespace

  • The URI namespace for the resources being linked is declared using the void:uriSpace property. Note that the object of the triple is a literal, not a URI.

Resource Type

  • The type of the resource being linked is declared with the dcterm:subject predicate.

In order to support applications using the linkset to understand and present the data, the following additional metadata should be included:

Title

  • The title of the dataset using dcterm:title predicate.

Description

  • A description of the dataset using dcterm:description predicate.

Links to further information

  • A link to the homepage for the dataset foaf:hompage.
  • Other pages that discuss the dataset may be included with the foaf:page relationship.

Data source

  • If the dataset contains distinguishable subsets, e.g. ChEMBL Compounds or ChemSpider to ChEMBL links, then these should be declared as subsets of the main dataset using the void:subset predicate. The metadata that is common between the datasets can be declared for the parent only.

In the VoID note [[VOID]], the void:subset propterty is used for both subset of and has subset. The declared semantics for the property is has subset. Within OPS, the property must be used with the has subset semantics. Therefore, if a linkset appears as a subset of a dataset, then it should be declared in the dataset metadata.

Other metadata that can be associated with a dataset may be included, see the RDF publishing guidelines [[OPS-RDF]], the VoID specification [[VOID]], and the provenance, authorship and versioning vocabulary [[PAV]] for additional metadata that may be incorporated. The dataset may also have a provenance graph associated with it providing more detailed information about the creation, authorship, and derivation of the dataset.

Examples

:chemspider
    a void:dataset ;
    dcterms:license <http://www.chemspider.com/Disclaimer.aspx> ;
    void:uriSpace "http://rdf.chemspider.com/" ;
    dcterms:subject <http://dbpedia.org/resource/Molecule> ;
    dcterms:title "ChemSpider"^^xsd:string ;
    dcterms:description "ChemSpider's Public Dataset"^^xsd:string ;
    dcterms:publisher <http://www.chemspider.com/> ;
    foaf:homepage <http://rdf.chemspider.com/> ;
    foaf:page <http://www.chemspider.com/> ;

In the example the homepage has been set to the RDF representation of ChemSpider as that is the dataset being described. The normal web interface has been given as an alternative web page.

The following example provides the metadata for the conversion of the ChEMBL database in RDF. This requires that provenance information about which version of ChEMBL has been converted are captured. The example also shows the use of subsets within the data.

:chembl-rdf
    a void:dataset ;
    dcterms:license <http://creativecommons.org/licenses/by-sa/3.0/> ;
    pav:importedFrom [
        foaf:homepage <http://www.ebi.ac.uk/chembl/> ;
        pav:version "13"^^xsd:integer ;
        dcterms:license <http://creativecommons.org/licenses/by-sa/3.0/> . ] ;
    pav:importedOn "2012-04-12"^^xsd:date ;
    pav:importedBy :egon ;
    void:uriSpace "http://data.kasabi.com/dataset/chembl-rdf"^^xsd:string ;
    dcterms:title "ChEMBL-RDF"^^xsd:string ;
    dcterms:description "The ChEMBL-RDF dataset"^^xsd:string ;
    foaf:homepage <http://kasabi.com/dataset/chembl-rdf> ;
    void:subset :chembl-rdf_compounds, :chembl-rdf_targets ;
    void:sparqlEndpoint <http://api.kasabi.com/dataset/chembl-rdf/apis/sparql>
    void:distinctSubjects "18018451"^^xsd:integer .
    
:chembl-rdf_compounds
    a void:dataset;
    void:uriSpace "http://data.kasabi.com/dataset/chembl-rdf/molecule/"^^xsd:string ;
    dcterms:subject <http://dbpedia.org/resource/Molecule> .
    
:chembl-rdf_targets
    a void:dataset;
    void:uriSpace "http://data.kasabi.com/dataset/chembl-rdf/target/"^^xsd:string ;
    dcterms:subject <http://dbpedia.org/resource/Protein> .

The above example shows the use of a blank node for stating where the data has been imported from. This could alternatively be captured through the declaration of another dataset containing the appropriate properties. It also assumes the existence of a local description of the concept :egon.

Linkset Metadata

A linkset is itself a dataset, and as such should provide the metadata one would associate with a dataset, as stated above. Specifically, the following metadata must be included:

Type

  • Declare that we have a linkset using void:linkset.

The void:linkset predicate is a sub-property of the void:dataset property.

License

  • Provide details of the licence under which the linkset is published using the dcterms:license property.

The license under which a linkset is published may be different from that of the dataset.

Linkset Provenance

  • If the linkset is a subset of another dataset, then this is captured with the void:subset predicate, with the linkset URI as the object of the triple.
  • The linkset authorship, i.e. the agent that generated the intellectual knowledge, is captured with the following properties:
    • pav:authoredBy to capture who authored the links. Currently we recommend the use of FOAF [[FOAF]] for the objects of these triples, see note above about future use of ORCID identifiers.
    • pav:authorOn to capture when the links were generated.
  • The linkset creator, i.e. the entity responsible for converting the links to the linkset format, is captured with the following properties:
    • pav:createdBy to capture who converted the links into the current digital format. Currently we recommend the use of FOAF [[FOAF]] for the objects of these triples, see note above about future use of ORCID identifiers.
    • pav:createdOn to capture when the links were converted to their digital form.

Additional information about how the links were created should be stated. For example, if the links were created using a tool such as Silk [[SILK]]. The links may also contain provenance information represented using PROV-O [[PROV-O]] and captured in a separate provenance graph.

Linkset Metadata

  • A declaration must be made about the datasets that are linked using the void:subjectsTarget and void:objectsTarget properties.
  • The link predicate is declared using void:linkPredicate.
  • The number of links in the linkset is declared using void:triples. This allows for validation that the entire linkset has been successfully loaded.

In order to support applications using the linkset to understand and present the data, the following additional metadata should be included:

Title

  • The title of the linkset using dcterm:title predicate.

Description

  • A description of the linkset using dcterm:description predicate.

Note that for linksets there is unlikely to be a web page directly dedicated to the linkset, as such that part of the dataset metadata is not stated as a requirement.

Example

The following example is the metadata associated with a link connecting a ChemSpider compound with a ChEMBL molecule. It reuses the dataset descriptions given in the examples above.

:chemspider_chembl-rdf_compounds
    a void:Linkset ;
    dcterms:license <http://www.opendatacommons.org/licenses/odbl/> ;
    pav:authoredOn "2012-02-22"^^xsd:date ;
    pav:authoredBy :egon ;
    pav:createdOn "2012-05-29"^^xsd:date ;
    pav:createdBy :christian ;
    void:subjectsTarget :chemspider ;
    void:targetsTarget :chembl-rdf_compounds ;
    void:linkPredicate skos:exactMatch ;
    void:triples "1015111"^^xsd:integer ;
    dcterms:title "ChEMBL-RDF Compounds ChemSpider Linkset"^^xsd:string ;
    dcterms:description "Linkset relating ChEMBL compounds to ChemSpider compounds."^^xsd:string .
    
:chembl-rdf_compounds void:subset :chemspider_chembl-rdf_compounds .
